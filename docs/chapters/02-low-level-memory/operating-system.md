# Операционная система

До сих пор вы потратили довольно много времени, работая очень близко к аппаратному обеспечению. Мы изначально обещали рассмотреть операционную систему. Теперь пришло время обсудить, как разработчики операционной системы очень серьезно отнеслись ко всем этим аппаратным ограничениям.

Из-за операционной системы и аппаратной архитектуры пределы физической памяти варьируются от 2 ГБ до 24 ТБ. При этом типичное коммерческое оборудование сегодня оснащено десятками гигабайт памяти. Если бы данная программа должна была использовать физическую память напрямую, ей пришлось бы самостоятельно управлять всеми созданными и удаляемыми регионами памяти. Такая логика управления памятью была бы не только сложной, но и повторялась бы в каждой программе. Кроме того, с точки зрения низкоуровневого программирования, использование памяти таким образом было бы громоздким. Каждая программа должна была бы запоминать, какие регионы памяти она использует, чтобы программы не мешали друг другу. Аллокаторы должны были бы взаимодействовать для надлежащего управления созданными и удаленными объектами. Это также довольно опасно с точки зрения безопасности: без какого-либо промежуточного уровня программа могла бы получить доступ не только к своим собственным регионам памяти, но и ко всем данным других процессов.

## Виртуальная память

Таким образом, была введена очень удобная абстракция — виртуальная память. Она переносит логику управления памятью в операционную систему, которая предоставляет любой программе так называемое адресное пространство. В частности, это означает, что каждый процесс считает себя единственным запущенным в системе и что вся память доступна для его собственных целей. Что еще лучше, поскольку адресное пространство виртуальное, оно может быть больше физической памяти. Кроме того, физическая оперативная память DRAM может быть расширена за счет вторичного хранилища, такого как жесткие диски для массового хранения, используемые файлом подкачки — об этом позже.

??? note "Примечание"

    Существуют ли операционные системы без виртуальной памяти? Для любого массового использования — нет. Но да, существуют очень маленькие операционные системы и фреймворки для встраиваемых систем, такие как ядро μClinux.

Менеджер памяти операционной системы имеет две основные задачи:

  *   **Сопоставление виртуального адресного пространства с физической памятью:** Виртуальные адреса имеют 32 бита на 32-битных машинах и 64 бита на 64-битных машинах (хотя в настоящее время используются только младшие 48 бит, что все равно позволяет получить адресное пространство размером 128 ТБ).
  *   **Перемещение некоторых регионов памяти из физической DRAM в жесткие диски:** Поскольку общий объем используемой памяти может превышать объём физической памяти, должен существовать механизм для обмена данными из DRAM и временного хранения их на более медленных носителях, таких как HDD или SSD. Эти данные хранятся в файле страниц (page file) или файле подкачки (swap file).

Перемещение фрагмента данных из оперативной памяти (RAM) для сохранения на временном хранилище явно связано с большим снижением производительности. Этот процесс в различных системах определяется как свопинг (swapping) или страничная подкачка (paging), главным образом по историческим причинам. Windows имеет специальный файл, называемый файлом страниц (page file), который хранит данные из памяти, отсюда и термин "страничная подкачка" (paging). Для Linux такие данные хранятся на выделенном разделе, называемом разделом подкачки (swap partition), откуда происходит термин "свопинг" (swapping) в Unix-подобных системах.

Виртуальная память реализована в ЦП (с помощью блока управления памятью — MMU) в сотрудничестве с операционной системой. Было бы слишком затратно сопоставлять виртуальное и физическое пространство байт за байтом, поэтому это делается с помощью непрерывных блоков памяти, называемых страницами. Также существует понятие таблица страниц (page directory), поддерживаемая операционной системой для каждого процесса, которая позволяет сопоставлять виртуальный адрес с физическим. Проще говоря, записи в таблице страниц указывают на начальные физические адреса страницы и другие метаданные, такие как привилегии.

В современных операционных системах распространенным подходом является введение многоуровневых таблиц. Это позволяет компактно хранить разреженное адресное пространство страниц при сохранении небольшого размера страницы. В настоящее время на большинстве архитектур типичный размер страницы составляет 4 КБ (включая x86, x64 и ARM) и используется четырехуровневая таблица страниц (см. [Рисунок 2-15](<#f-2-15>)).

Когда виртуальный адрес преобразуется в физический адрес, требуется обход таблицы страниц (page directory walk):

  *   Селектор первого уровня выбирает запись в директории первого уровня, которая указывает на одну из записей директории второго уровня.
  *   Селектор второго уровня выбирает запись в конкретной записи директории второго уровня, которая указывает на одну из записей директории третьего уровня.
  *   Селектор третьего уровня выбирает запись в конкретной записи директории третьего уровня, которая указывает на одну из записей директории четвертого уровня.
  *   В конечном итоге селектор четвертого уровня выбирает запись в конкретной записи директории четвертого уровня, которая напрямую указывает на некоторую страницу в физической памяти.
  *   Смещение (Offset) указывает на конкретный адрес внутри выбранной страницы.

Такой перевод требует обхода дерева каталога страниц, который хранится в физической памяти. Это означает, что он также может быть кэширован в кэшах L1/L2/L3. Но все равно это приведет к огромным накладным расходам, если каждое преобразование адреса (операция, выполняемая очень часто) потребует доступа к этим данным. Таким образом, были введены буферы трансляции Look-Aside (TLB), которые кэшируют сам перевод. Идея проста — TLB работает как карта, где селектор является ключом, а начало физического адреса страницы — значением. TLB созданы так, чтобы быть чрезвычайно быстрыми, поэтому они малы с точки зрения хранения. Они также являются многоуровневыми, чтобы имитировать структуру каталога страниц. Результатом промаха TLB (не кэшированного виртуального в физическое преобразование) является выполнение полного обхода каталога страницы, что является дорогостоящим.

??? note "Интересное примечание"

    Как и во всех кэшах, предварительная выборка в TLB является сложной задачей: если сам ЦП должен инициировать эту предварительную выборку (например, из-за предсказания ветвления), он может вызвать ненужный обход таблицы страниц (поскольку предсказание ветвления может оказаться неверным). Чтобы предотвратить это, предварительная выборка TLB контролируется программным обеспечением. Также следует помнить, что аппаратная предварительная выборка ограничена страницами. В противном случае она вызовет промах страницы, что станет большой ненужной накладной стоимостью, если догадка окажется неверной. Есть ли какие-либо актуальные оптимизации TLB, которые стоит учитывать в своем коде? Основная цель состоит в том, чтобы уменьшить количество страниц, чтобы таблица страниц оставалась небольшой, а это, в свою очередь, снизит количество промахов TLB. С точки зрения приложений .NET, большие страницы, описанные в следующем разделе, являются единственным способом повлиять на управление страницами.
    
    Как правило, L1 оперирует виртуальными адресами, поскольку стоимость их преобразования в физические адреса была бы намного выше, чем сам доступ к кешу. Это означает, что когда страница изменяется, все или некоторые строки кеша должны быть инвалидированы. Следовательно, большое количество изменений страниц негативно влияет на производительность кэша.

## Большие страницы

Как мы видели ранее, преобразования виртуальных адресов могут быть затратными, и было бы очень хорошо избегать их как можно чаще. Одним из подходов является использование более крупного размера страницы. Это потребует меньше преобразований адресов, поскольку многие адреса поместятся на одну страницу, при этом преобразование уже закэшировано в TLB. Однако большие страницы могут привести к неэффективному использованию ОЗУ или дискового пространства, используемого файлом подкачки. Существует одно решение — так называемые крупные (или гигантские) страницы. При аппаратной поддержке они позволяют создавать большой непрерывный блок физической памяти, состоящий из многих последовательно размещенных обычных страниц. Эти страницы обычно в два-три раза больше по объему, чем обычная страница. Они могут быть полезны в сценариях, когда программа требует произвольного доступа к гигабайтам данных. Механизмы баз данных являются примерами таких потребителей больших страниц. Операционная система Windows также отображает свои основные образы и данные ядра с использованием крупных страниц. Большая страница не подлежит страничной подкачке (не может быть перемещена в файл страниц) и поддерживается как в Windows, так и в Linux. К сожалению, заполнить большую страницу довольно трудно: это приводит к фрагментации. Кроме того, может не быть доступного непрерывного диапазона физической памяти.

Крупные страницы можно включить в .NET начиная с версии .NET Core 3.0 с помощью настройки COMPlus_GCLargePages/DOTNET_GCLargePages. Это будет описано более подробно в Главе 11.

## Фрагментация виртуальной памяти

Как всегда, когда дело доходит до выделения и освобождения памяти, фрагментация может проявиться, как было упомянуто в главе 1 при обсуждении концепции кучи. В случае виртуальной памяти это означает, что операционная система не сможет зарезервировать непрерывный блок памяти заданного размера в адресном пространстве процесса. Между уже зарезервированной памятью нет достаточно большого промежутка, хотя суммарный размер всех свободных промежутков может значительно превышать требуемый размер.

Эта проблема может стать критической для 32-битных приложений, поскольку их виртуального пространства часто недостаточно для современных требований. Фрагментация проявляется особенно остро, когда процесс выделяет большие сегменты памяти и работает в течение длительного времени: это именно та ситуация, с которой приходится сталкиваться при работе веб-приложений на .NET в 32-битной версии (размещенных в IIS). Чтобы предотвратить фрагментацию, процесс должен надлежащим образом управлять памятью (и для процесса .NET это означает сам CLR). Мы подробно рассмотрим такие детали при описании алгоритмов сборки мусора в Главах 7–10, поскольку это требует более глубокого понимания самого фреймворка .NET.

## Общая компоновка памяти

Зная базовый блок управления памятью, мы можем перейти к обсуждению памяти на более высоком уровне. Какова планировка памяти программы? При описании типичной планировки памяти часто используется такая схема, как показано на [Рисунке 2-16](#f-2-16). На ней представлено типовое адресное пространство программы.

Как видно, виртуальное адресное пространство разделено на две области:

*   **Пространство ядра (Kernel space):** Верхний диапазон адресов занят самой операционной системой. Оно называется пространством ядра, поскольку именно здесь отображаются драйверы устройств и структуры данных ядра. Обратите внимание, что доступ к этой части адресного пространства имеют только коды ядра.
*   **Пользовательское пространство (User space):** Нижняя часть адресного пространства доступна коду приложения (за исключением самой нижней области, которая используется для обнаружения нулевых указателей).

С вашей точки зрения, разумеется, интерес представляет только пользовательское пространство: именно здесь CLR хранит ваши данные. Благодаря механизму виртуальной памяти каждый процесс имеет доступ ко всему адресному пространству — как будто он единственный процесс в системе.

Вот области памяти, видимые на [Рисунке 2-17](<#f-2-17>):

*   Для каждого потока создается стек.
*   Операционная система предоставляет различные API для среды выполнения, позволяющие создавать кучи (подробнее об этом позже).
*   Когда запускается приложение, операционная система отображает соответствующий двоичный файл с диска в адресное пространство. Некоторые данные только для чтения могут быть разделены между процессами, например, исполняемый ассемблерный код.
*   Такая схема полезна для получения общего представления о планировке адресного пространства. Однако реальность сложнее. Давайте рассмотрим контекст двух основных операционных систем, поддерживаемых средой выполнения .NET: Windows и Linux.

??? note "Примечание"

    Существует механизм под названием Address Windowing Extensions (AWE), который позволяет получить доступ к большему объему физической памяти, чем указано здесь, а затем отобразить только части этого объема в виртуальном адресном пространстве с помощью «Окна AWE» (AWE Window). Это может быть особенно полезно в 32-битной среде для преодоления ограничения в 2 или 3 ГБ на процесс. Однако это неактуально в данном контексте, поскольку CLR не использует эту функцию.

Ограничения размера виртуальной памяти одного процесса стали особенно острыми к концу эпохи 32-битных систем. Ограничение до 2 ГБ (или 3 ГБ в расширенном режиме) может представлять собой серьезную проблему для крупномасштабных корпоративных приложений. Классическим примером является веб-приложение ASP.NET, размещенное в IIS на машинах Windows Server 32-битной архитектуры. Если этот лимит достигался, не оставалось иного выбора, кроме как перезапускать все веб-приложение целиком. Это вынуждало к горизонтальному масштабированию больших веб-систем: созданию множества экземпляров серверов, каждый из которых обрабатывал меньший объем трафика и, следовательно, потреблял меньше памяти. Сегодня мир доминируют 64-битные системы, и ограниченное виртуальное адресное пространство больше не является проблемой. Однако следует отметить, что программа, скомпилированная в 32-битном режиме, по-прежнему имеет ограничение виртуального адресного пространства в 4 ГБ даже на серверах Windows 64-битной архитектуры.

Менеджер памяти Windows предоставляет два основных API:

*   **Виртуальная память (Virtual memory):** Это низкоуровневый API, который работает непосредственно с адресным пространством для резервирования и фиксации страниц. Функции `VirtualAlloc` и `VirtualFree` используются средой выполнения CLR.
*   **Куча (Heap):** Более высокоуровневый API, предоставляющий Аллокатор (вспомните его из Главы 1), который используется средой выполнения C и C++ для реализации функций `malloc`/`free` и `new`/`delete`. Этот уровень включает, помимо прочего, функции `HeapAlloc` и `HeapFree`.

Поскольку CLR имеет собственную реализацию Аллокатора для создания объектов .NET (что вы подробно увидите в Главе 6), используются только виртуальные API. Короче говоря, CLR запрашивает у операционной системы дополнительные страницы, а соответствующее выделение объектов внутри этих страниц обрабатывается самой средой выполнения. CLR создает свои собственные внутренние нативные структуры данных с помощью оператора `new` языка C++ (который, в свою очередь, построен поверх API Кучи/Heap).

В Windows API виртуальной памяти работает в два этапа. Во-первых, вы резервируете непрерывный диапазон памяти в адресном пространстве, вызывая VirtualAlloc с MEM_RESERVE и PAGE_READWRITE (поскольку вы хотите иметь возможность хранить и считывать данные оттуда) в качестве параметров. Диспетчер памяти Windows возвращает адрес, с которого начинается зарезервированный диапазон памяти в адресном пространстве. Будьте внимательны: вы пока не можете получить доступ к этой памяти! Во-вторых, когда вам нужно прочитать или записать в эту память, вы должны зафиксировать нужные страницы из этого зарезервированного диапазона. Вам не нужно фиксировать весь диапазон, только нужные страницы. Именно этим и занимается среда CLR: резервирует большие диапазоны в адресном пространстве один раз, а затем фиксирует страницы, когда требуется место для выделения дополнительных объектов. При первом обращении к зафиксированной странице диспетчер памяти гарантирует, что она инициализирована нулевыми строками.

Когда объекты собираются и обнаруживается достаточное количество свободного места, соответствующие страницы декоммитятся путем вызова VirtualFree с параметром MEM_DECOMMIT, чтобы вернуть память диспетчеру памяти Windows. Точные детали объясняются в главе 10.

Следующий вопрос, который вы, возможно, захотите задать, — как измерить потребление «памяти» вашими приложениями. Исходя из того, что уже было объяснено, память, в которой хранятся ваши объекты, называется частной выделенной памятью вашего процесса. Общая выделенная память подсчитывает данные только для чтения, такие как ассемблерный код, хранящийся в исполняемых файлах, и не представляет интереса в данном контексте.

Более того, приватные страницы также могут быть заблокированы, что заставляет их оставаться в физической памяти (они не будут перемещены в файл подкачки), пока они явно не разблокируются или пока приложение не завершит работу. Блокировка может быть полезной для критичного для производительности пути выполнения программы. Мы рассмотрим пример использования блокировки страниц в пользовательском хосте CLR, показанный в главе 15.

Зарезервированные и выделенные страницы управляются процессом с помощью вышеупомянутых вызовов методов VirtualAlloc/VirtualFree и VirtualLock/VirtualUnlock. Также стоит отметить, что попытка доступа к свободной или зарезервированной памяти приведет к исключению нарушения доступа (Access Violation Exception), поскольку эта память еще не может быть отображена на физическую память.

??? note "Примечание"

    Зачем кто-то придумал такой двухэтапный процесс получения памяти? Как упоминалось ранее, последовательная модель доступа к памяти хороша по многим причинам. Пространство, состоящее из непрерывной последовательности страниц, предотвращает фрагментацию и таким образом оптимизирует использование TLB (таблицы переводов страниц) и избегает обходов директорий страниц. Непрерывная память, конечно же, также выгодна для использования кэша. Поэтому хорошо зарезервировать какое-то большее пространство заранее, даже если нам оно сейчас не нужно. Именно так обычно реализуются стеки потоков операционной системой: весь стек резервируется в адресном пространстве, а страницы выделяются одна за другой при выполнении более глубоких вызовов методов, требующих больше места для параметров и локальных переменных.

На [рисунке 2-17](<#f-2-17>) графически изображены отношения между различными «наборами памяти» в виде перекрывающихся наборов:

  * **Working set** \- Рабочий набор: Это часть виртуального адресного пространства, которая в настоящее время находится в физической памяти. Его можно разделить на несколько частей

  * **Private working set** \- Приватный рабочий набор: Состоит из зафиксированных (приватных) страниц в физической памяти

  * **Shared working set** \- Общий рабочий набор: Состоит из страниц, которые являются общими для других процессов

  * **Private bytes** \- Приватные байты: Все зафиксированные (приватные) страницы – как в физической, так и в выгружаемой памяти

  * **Virtual bytes** \- Виртуальные байты: зарезервированная память в адресном пространстве (как зафиксированная, так и незафиксированная)

  * **Paged bytes** \- Выгружаемые байты: Часть виртуальных байтов, хранящихся в файле подкачки

Довольно сложно, не так ли? Возможно, теперь вы понимаете, что ответ на вопрос «сколько памяти фактически занимает наш процесс .NET» не так очевиден. На какой из этих показателей следует смотреть? Ошибочно полагать, что самым важным показателем является приватный рабочий набор, поскольку он показывает фактическое влияние процесса на потребление наиболее важной физической оперативной памяти (RAM). Однако в случае утечки памяти будут расти именно приватные байты, а не обязательно приватный рабочий набор. Вы узнаете, как отслеживать эти показатели, в следующей главе. Вы также поймете, что фактически отображается Диспетчером задач в столбце «Память» процесса.

Из-за своих внутренних структур при резервации Windows области памяти для адресного пространства процесса учитывается следующее ограничение: как начало области, так и её размер должны быть кратны размеру страницы системы (обычно 4 КБ) и так называемой гранулярности выделения (обычно 64 КБ). На практике это означает, что каждый начальный адрес зарезервированной области и её размер кратны 64 КБ. Если вы хотите выделить меньше, остаток будет недоступным (неиспользуемым). Таким образом, правильное выравнивание и размер блоков имеют решающее значение для избежания потери памяти.

Хотя вы не управляете памятью на уровне Virtual API в повседневной работе, эти знания могут помочь вам понять важность выравнивания в коде CLR. Внимательный читатель может спросить, почему гранулярность выделения составляет 64 КБ, а размер страницы — 4 КБ. Рэймонд Чен, сотрудник Microsoft, ответил на этот вопрос в 2003 году [Почему гранулярность выделения адресного пространства равна 64К? – [devblogs.microsoft.com](https://devblogs.microsoft.com/oldnewthing/20031008-00/?p=42223). И как обычно в таких случаях, ответ очень интересен. Гранулярность выделения обусловлена в основном историческими причинами. Ядро всей современной семьи операционных систем восходит к корням раннего ядра Windows NT. Оно поддерживало ряд платформ, включая архитектуру DEC Alpha. И именно из-за разнообразия поддерживаемых платформ было введено такое ограничение. А поскольку оно не оказалось неудобным для других платформ, предпочли иметь общую базовую кодовую основу ядра. Более подробные объяснения вы найдете в упомянутой статье.

## Структура памяти в Windows

Теперь давайте подробнее рассмотрим процесс .NET, запущенный в Windows. Процесс содержит одну кучу по умолчанию (в основном используемую внутренними функциями Windows) и любое количество дополнительных куч (создаваемых с помощью Heap API). Один из примеров — куча, создаваемая средой выполнения Microsoft C и используемая операторами C/C++, как упоминалось ранее. Существует три основных типа куч:

  *   **Обычная куча (Normal/NT heap):** Используется обычными приложениями (не из Универсальной платформы Windows – UWP). Она обеспечивает базовую функциональность управления блоками памяти.

  *   **Куча с низкой фрагментацией (Low-fragmentation heap):** Дополнительный уровень поверх обычной функциональности кучи, который управляет выделением в заранее определенных блоках различных размеров. Это предотвращает фрагментацию для небольших данных и дополнительно ускоряет их доступ благодаря внутренним оптимизациям ОС.

  *   **Сегментная куча (Segment heap):** Используется приложениями Универсальной платформы Windows, которые предоставляют более сложные аллокаторы (включая низкофрагментированный аллокатор, подобный упомянутому выше).

Как упоминалось ранее, планировка адресного пространства процесса разделена на две части: верхние адреса заняты ядром, а нижние — пользователем (программой). Это показано на [Рисунке 2-18](#f-2-18) (32-битная архитектура слева, 64-битная справа). На машинах с 32-разрядной архитектурой, в зависимости от значения флага большого адреса, пользовательское пространство занимает нижние 2 или 3 ГБ. В современных 64-битных ЦП, поддерживающих 48-битное адресование, для пользовательского и ядерного пространства доступно по 128 ТБ виртуальной памяти (в предыдущих версиях — Windows 8 и Server 2012 — это было 8 ТБ).

С некоторой степенью приближения типичная структура пользовательского пространства программы .NET на Windows выглядит следующим образом:

  * Упомянутая ранее куча по умолчанию.

  * Большинство образов (exe, dll) расположены по высоким адресам.

  * Стеки потоков (упомянутые в предыдущей главе) могут располагаться где угодно. Каждый поток в процессе имеет свою область стека потока. Это включает потоки CLR, которые просто оборачивают нативные системные потоки.

  * Кучи GC, управляемые CLR для хранения создаваемых вами объектов .NET (это обычные страницы в терминологии Windows, зарезервированные и выделенные через Virtual API).

  * Различные приватные кучи CLR для внутренних целей. Мы рассмотрим их более подробно в следующих главах.

  * Конечно, есть также довольно много свободного виртуального адресного пространства, включая огромные блоки порядка гигабайт и терабайт (в зависимости от архитектуры) где-то посередине виртуального адресного пространства.

Начальный размер стека потока на Windows (как зарезервированный, так и изначально выделенный) берется из заголовка исполняемого файла (обычно известного как файл EXE). Для вручную созданных потоков размер стека также можно указать путем ручного вызова API CreateThread Windows.

То, как среда выполнения .NET вычисляет размер стека по умолчанию, довольно сложно. Значение по умолчанию составляет 1 МБ для типичной 32-битной компиляции и 4 МБ для типичной 64-битной компиляции. Данные стека довольно малы, а стек вызовов обычно довольно неглубок (сотни вложенных вызовов встречаются редко). Это делает 1 или 4 МБ хорошим значением по умолчанию.

Однако если вы когда-либо сталкивались с исключением StackOverflowException, значит вы столкнулись с этим ограничением. Даже в этом случае это, скорее всего, вызвано ошибкой программирования, например бесконечной рекурсией. Если по какой-то причине вы пишете программу, которой нужно хранить много данных в стеке, вы можете изменить заголовок бинарного файла. Исполняемые файлы .NET интерпретируются как обычные исполняемые файлы, поэтому это изменение будет учтено операционной системой. Мы увеличим этот лимит размера стека для такой цели в главе 4.

??? note "Примечание"

    По соображениям безопасности был введен механизм рандомизации адресного пространства (ASLR), который делает все макеты, показанные на [рисунке 2-18](<#f-2-18>), только иллюстративными. Это приводит к тому, что все компоненты (двоичные изображения) размещаются случайным образом по всему адресному пространству, чтобы не повторять какой-либо общий шаблон, который может быть использован злоумышленником.

Мы надеемся, что такой общий обзор позволит вам лучше понять управление памятью CLR в контексте всей экосистемы Windows. Мы еще раз обратимся к этим знаниям при детальном описании структуры памяти CLR.

## Управление памятью в Linux

До недавнего времени любая ссылка на Linux в книге о .NET ограничивалась бы проектом Mono. Но времена изменились. С появлением среды .NET Core (включая .NET 5+) больше невозможно игнорировать не-Windows системы. Более того, популярность запуска .NET на машинах без Windows продолжает расти вместе с распространением Linux в мире контейнеров. Мы уделим много внимания реализации среды выполнения .NET Core и .NET 5+. Поскольку Linux использует ту же аппаратную технологию, включая страницы, MMU и TLB, большая часть знаний уже охвачена описаниями в предыдущих подразделах. Здесь мы сосредоточимся только на интересных различиях. Поскольку все больше людей придется развертывать приложения в этой не-Windows среде .NET, очень полезно также понимать основы Linux.

Популярные и наиболее часто используемые дистрибутивы операционных систем Linux также используют концепцию виртуальной памяти. Их лимиты на процесс также очень похожи и обобщены в [таблице 2-5](<#t-2-5>).

*Каноническая 48-битная адресация

Как и в Windows, базовым строительным блоком в Linux является страница, и её размер также обычно составляет 4 КБ. Страница может находиться в одном из трех различных состояний, перечисленных ниже:

  * Свободная (Free): Не назначена ни одному процессу, ни самой системе.

  * Выделенная (Allocated): Назначена процессу.

  * Общая (Shared): Зарезервирована для процесса, но может быть разделена с другими процессами. Это обычно относится к бинарным образам и отображенным в памяти файлам системных библиотек и ресурсов.

Это обеспечивает более простое и понятное представление о потреблении памяти процессом по сравнению с операционной системой Windows. Как видно, по сравнению с Windows отсутствует этап неявной резервации страниц, хотя он все еще существует явно. В Linux встроен механизм ленивого выделения, который берет это на себя. Когда вы выделяете память в Linux, она считается выделенной, но физические ресурсы не назначаются (следовательно, это похоже на резервацию в Windows). Фактическое назначение ресурсов (потребление физической памяти) не произойдет до тех пор, пока они действительно не понадобятся при доступе к этой конкретной области памяти. Если вы хотите про активно подготовить такие страницы в сценариях, критичных для производительности, вы можете «коснуться» их, обращаясь к их памяти, например, читая хотя бы один байт.

Зная возможные состояния страниц, вы можете посмотреть, на какие категории делится память процесса в Linux. Вокруг этого существует довольно много путаницы. Многие инструменты на базе Linux говорят немного разные вещи об этой теме. Использование памяти процессом можно измерить с точки зрения следующих терминов:

  * Виртуальная (отмечается некоторыми инструментами как vsz): Общий размер виртуального адресного пространства, зарезервированного на данный момент процессом. В популярном инструменте "top" он показан в столбце VIRT.

  * Резидентная (Resident Set Size, RSS): Пространство страниц, которые в данный момент находятся в физической памяти. Некоторые резидентные страницы могут быть общими между процессами (например, страницы с файловой поддержкой). Следовательно, это эквивалент «рабочего набора» на платформе Windows. В "top" это называется столбцом RES. Его можно дополнительно разделить на:

    * Приватные резидентные страницы: Это все анонимные резидентные страницы, зарезервированные для этого процесса (указываются счетчиком ядра MM_ANONPAGES). Они несколько соответствуют «приватному рабочему набору» в Windows.

    * Общие резидентные страницы: Это как страницы с файловой поддержкой (указываются счетчиком ядра MM_FILEPAGES), так и анонимные резидентные страницы процесса, соответствующие «общему рабочему набору». В "top" это называется памятью SHR.

  * Приватная: Все приватные страницы процесса. В инструменте "top" это столбец DATA. Обратите внимание, что это индикатор зарезервированной памяти и не говорит о том, сколько из неё уже было доступно («было касание») и стало резидентной. Это соответствует «приватным байтам» в Windows.

  * Swap: Часть виртуальной памяти, которая была сохранена в файле подкачки (swap file).

## Структура памяти в Linux

Структура памяти процесса Linux очень похожа на представленную для Windows. Для 32-битной версии пользовательское пространство составляет 3 ГБ, а пространство ядра — 1 ГБ. Эта точка разделения может быть изменена с помощью параметра CONFIG_PAGE_OFFSET, который настраивается во время сборки ядра. Для 64-битной версии разделение происходит по похожему адресу, как и в Windows (см. [рисунок 2-20](<#f-2-20>)).

Аналогично Windows, система предоставляет API для работы со страницами памяти. Он содержит:

  * **mmap**: Для прямой манипуляции страницами (включая файловые отображения, разделяемые и обычные, а также анонимные отображения, которые не связаны ни с каким файлом, но используются для хранения данных программы).

  * **brk/sbrk**: Это ближайший эквивалент метода VirtualAlloc. Он позволяет устанавливать/увеличивать так называемый «программный break», что фактически означает увеличение размера кучи.

Известные аллокаторы C/C++ используют `mmap` или `brk` в зависимости от размера выделения. Этот порог можно настроить с помощью функции `mallopt` и параметра `M_MMAP_THRESHOLD`. Как вы увидите позже, CLR использует `mmap` с анонимными приватными страницами.

Существует одно существенное различие в обработке стека потока между Linux и Windows. Поскольку нет двухэтапной резервации памяти, стек просто расширяется по мере необходимости. Нет предварительной резервации соответствующих страниц памяти. И поскольку следующие страницы создаются по мере необходимости, стек потока не является непрерывной областью памяти.

## Влияние операционной системы

Есть ли какие-либо различия в управлении памятью, которые были учтены в кроссплатформенной версии сборщика мусора (Garbage Collector), включенного в CLR? В целом код GC очень платформно-зависим, но по очевидным причинам на каком-то этапе необходимо делать системные вызовы. Менеджер памяти в обеих операционных системах работает схожим образом — он основан на виртуальной памяти, пейджинге и похожем способе выделения памяти. Хотя, конечно, вызываемые системные API различаются, концептуально нет специфических различий в коде, за исключением двух ситуаций, которые мы хотели бы описать сейчас.

Первое отличие уже упоминалось. В Linux нет двухэтапного способа резервирования и последующего выделения памяти. В Windows вы можете использовать системный вызов, чтобы сначала зарезервировать большой блок памяти. Это будет создание соответствующих системных структур без фактического использования физической памяти. Только при необходимости выполняется второй этап выделения диапазона памяти. Поскольку в Linux нет этого механизма, память может быть выделена только без «резервирования». Однако для имитации двухэтапного способа резервирования/выделения потребовался системный API. Для этой цели использовался популярный трюк. В Linux «резервирование» осуществляется путем выделения памяти с режимом доступа PROT_NONE, что фактически означает, что доступ к этой памяти запрещен. Однако в такой зарезервированной области вы можете затем снова выделить определенные под регионы с обычными правами, таким образом имитируя «выделение» памяти.

Второе различие — это так называемый механизм наблюдения за записью в память (memory write watch). Как вы увидите в последующих главах, Сборщику мусора необходимо отслеживать, какие области памяти (страницы) были изменены. Для этой цели Windows предоставляет удобный API. При выделении страницы можно установить флаг MEM_WRITE_WATCH. Затем, используя API GetWriteWatch, можно получить список измененных страниц. Во время реализации .NET Core стало очевидно, что в Linux нет системного вызова для создания такого механизма наблюдения. По этой причине эту логику пришлось переместить в барьер записи (механизм, подробно объясненный в главе 5), который поддерживается средой выполнения без какой-либо поддержки со стороны операционной системы.
